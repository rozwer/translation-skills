# 多ジャンル翻訳支援スキルシステム — 仕様書 (SPEC)

> 対象業務: 翻訳・ナレーション（成果物多岐：アナウンス台本, 科研申請書 …etc。主に日本語→英語、およびクライアントが作成した英訳(AI使用)の修正。日本語ナレ原稿そのものの推敲も含む）
> 実行基盤: Claude Code（Claude Desktop アプリの Code タブ）。Pro プランのためトークン使用量の節約も重要。
> 実行前提: 使用者は英語翻訳のプロだが、AI 使用は完全な初学者。

---

## 0. このドキュメントについて

本書は「ジャンルごとに最適な処理を自動で選び、使うほど会社専用に育っていく翻訳支援システム」の設計仕様である。

**状態の明示（重要）**
- **確定済み**: 全体アーキテクチャ、実行環境、各コンポーネントの役割と既存の足場、ツールの適性マップ。
- **本改訂で確定した設計判断**: ルーターの判定方式（§2: 明示選択を主経路とする）、learnings 昇格の二経路（§4）、機密マスキングの強制タイミング（§3.5 / §4）、検出系の運用形態（§6: 手動チェックリスト化）、外部ツール採用の判定基準（§8.1）、育成の単一正本運用（§7）。
- **未確定（§9 TODO）**: 各プロファイルのドメイン content、運用者の OS、足場の採用 or 自作の最終判定、共有プラン、境界ケースの分類規約。

## 1. 全体設計

### 1.1 実行環境の確定根拠

中核は **Claude Code（Desktop の Code タブ）**。理由は、本システムの肝である「スキルの自己更新ループ」が成立するのに、書き込み可能なファイルシステムが必須だから。

- Code は `~/.claude/skills/` やプロジェクトの `.claude/skills/` を直接読み書きし、git にコミットできる → 更新ループが native に閉じる。
- Code タブは GUI（視覚的チャット・ファイルツリー・差分レビュー・履歴）であり、ターミナル不要。非開発者でも運用可能。
- **差分レビュー UI が、設計上の「承認ゲート」をそのまま提供する**（更新案を人が差分で確認して承認）。
- auto-memory（ユーザーの修正に基づき Claude 自身がメモを書く機構）が標準装備で、「指摘を記憶」の土台になる。auto-memory と `learnings.md` の役割分担は §4.0 で定義する。
- 同一サブスク・同一アプリ内に Chat / Cowork / Code の 3 タブ。日々の翻訳は Cowork タブで「投げて離席」も可能。

### 1.2 環境ごとの自動度（設計上の制約）

| 環境 | 更新ループ | 備考 |
|---|---|---|
| Claude Code（正本） | **完全自動で閉じる** | 書込可 FS＋git。承認ゲートは設計選択として残す |
| Claude.ai Web | 半自動 | 生成・編集は可能だが、ライブラリ反映は「Edit with Claude」等の明示ステップ |
| Cowork | 条件付き | Project 作業ファイルは書込可・永続。ただし下記の原則必須 |

**最重要原則**: **育てる対象は「書き込み可能なファイル」（Project 作業ファイル / `.claude/skills/` 内の .md）に限る。** Cowork VM 内の「インストール済みスキル」はインストール時のキャッシュ・スナップショットで読み取り専用であり、セッション中の自己書き換えは不可（再インストールが必要）。ここに育てると揮発・サイロ化する。

### 1.3 二層アーキテクチャ

- **親ルーター**（オーケストレーター）: インタビュー → レジスタ判定 → 該当パイプライン呼び出し → 最後に「指摘のルール化提案」を出す。
- **ジャンル別子スキル**: レジスタ・プロファイルごとに、忠実さ基準・使う書き換えツール・チェック手順・NG 集を保持。
- **共通レイヤー**: どのプロファイルにも属する横断ルールと、境界ケースの指摘の置き場。

合成（スキルが別スキルを呼ぶ）と progressive disclosure を前提とする。子スキルは肥大させず、単一責務で分割する。

### 1.4 リポジトリ構成（提案）

```
translation-skills/
├── SPEC.md                      # 本書（仕様の正本。変更はここに集約）
├── README.md                    # 導入・運用手順のみ。仕様は SPEC.md へリンクし重複させない（§M2 対策）
├── router/
│   └── SKILL.md                 # 親ルーター（インタビュー＋振り分け＋ルール化提案）
├── profiles/                    # レジスタ・プロファイル＝ジャンル別子スキル
│   ├── audio-script/SKILL.md    #  音声台本
│   ├── editorial/SKILL.md       #  広報・読み物
│   ├── academic-grant/SKILL.md  #  学術・申請
│   ├── contract-legal/SKILL.md  #  契約・公文書
│   └── etc...
├── common/
│   └── house-style.md           # 横断ハウススタイル／共通 NG／境界ケースの置き場
├── tools/                       # 外部スキルの取り込み・配線
│   ├── rewrite/                 #  書き換え系（avoid-ai-writing 等）
│   └── detect/                  #  検出系の運用メモ（手動チェックリスト）
├── improvement/
│   ├── SKILL.md                 # 更新スキル（learnings loop / 統治ルール）
│   └── learnings.md             # 蓄積される指摘ログ（承認待ちキュー＋確定ルール）
├── glossary/                    # クライアント別・分野別 用語集（匿名トークン管理は §3.5）
└── .gitignore                   # 機密原稿は追跡しない（§7）
```

> **README の責務（§M2 対策）**: README には「導入手順・起動方法・日々の使い方」だけを書き、仕様の実体は SPEC.md に置く。仕様変更時に二重更新が発生して README が腐るのを防ぐ。README から該当 SPEC セクションへリンクする。

## 2. ルーター仕様

**役割**: タスクが来たとき、ツール起動前に「これは何のジャンルか／どのパイプラインで処理すべきか」を決定する。Claude の暗黙の推測（誤選択しやすい）を、**ユーザーの明示選択を主経路とする決定的トリアージ**に置き換える。

### 2.1 判定方式（§C1 対策：「決定論」の名と実を一致させる）

LLM にジャンルを推論させると、同一原稿でも判定がブレる。とくに境界ケースで顕著。そこで判定経路を二段に分ける。

- **主経路（決定的）**: 設問 1「成果物の種類」を**ユーザーの明示選択**とし、選択肢 ID をそのままプロファイル ID にマッピングする。LLM の推論を挟まない。同じ選択 → 必ず同じプロファイル。
- **フォールバック経路（推論あり）**: ユーザーが「その他／わからない」を選んだときに限り、LLM が原稿の特徴から候補プロファイルを**提示**する（自動確定しない）。最終決定は人が行う。
- 推論を使ったか否かを必ずログに記録し、フォールパック経由の判定は後追いで見直せるようにする。

### 2.2 入力（インタビュー設問）

1. **成果物の種類**（台本／広報／学術・申請／契約・公文書／その他）← この選択が決定的にプロファイルを決める
2. 忠実さの要求（意図に忠実でよい transcreation 可 ↔ 逐語・定訳厳守）
3. 制約（尺・文字数・定訳の有無・締切・分野）
4. 人間レビューの要否（クライアント要件・機密度）
5. **機密度タグ**（クライアント名・固有名詞を含むか／公開可否）← §3.5 のマスキング判断に使う

### 2.3 処理

- 設問 1 からプロファイルを確定（§2.1 主経路）。「その他」のみ §2.1 フォールバック。
- 該当する子スキルと、使用する書き換えツール（§5）を選択。
- **境界ケース**（台本と読み物の中間など）は、無理に片方へ寄せず、判定の不確実性を**明示フラグ**にして残し、`common/` レイヤーの規則で処理する（§5.5 の分類規約に従う）。
- 何を発火させるかを**実行前に宣言**し、jsonl で追記ログを残す（監査可能性）。ログには「主経路 or フォールバック」「機密度タグ」を含める。

**参考実装（足場）**: `hussi9/skill-router`（1 枚の SKILL.md で 3 問トリアージ、宣言＋jsonl ログ＋監査スクリプト）。自作せず構造を借りて、本用途で検証する（採用判定は §8.1 の基準で）。

## 3. レジスタ・プロファイル定義

本システムの分類単位。**各プロファイルの詳細規則は §9 TODO（ドメイン content）。** ここでは枠と方針のみ定義する。

| プロファイル | 忠実さの種類 | 書き換えツール | 検出の扱い | 人間レビュー | 自動化の深さ |
|---|---|---|---|---|---|
| 音声台本 | 効果・意図に忠実（transcreation 可） | blader/humanizer（声を足す） | 基本不要 | 軽め | 厚い |
| 広報・読み物 | 自然さ寄り・事実は固定 | avoid-ai-writing（徹底クリーニング） | 必要時のみ手動チェック（§6.2） | 中 | 中 |
| 学術・申請 | 命題に高忠実・定訳厳守 | humanizer_academic（formality 保持） | **本流に組み込まない**（§6.3） | **必須** | 浅い（下訳＋整合まで） |
| 契約・公文書 | 逐語・改変厳禁 | **書き換え禁忌**（detect/監査のみ） | 逐語性最優先・**本流に組み込まない** | **必須** | 最小 |
| etc... | | | | | |

**プロファイル共通の作法**
- 各プロファイルは「忠実さ基準・使用ツール・チェック手順・NG 集」を持つ。
- 音声台本系は「声に出して読めるか」を推敲基準に含める（尺・息継ぎ・難読語・数字の読み）。
- 学術・契約系は、自然さより忠実性を優先。claim の強弱を動かさない、定訳から外れない、存在しない参照を入れない。

### 3.5 機密マスキングの規約（§C3 対策：learnings/glossary 経由の漏洩を止める）

機密原稿そのものは `.gitignore` で追跡しない（§7）。しかし**原稿から抽出した指摘・用語は learnings.md / glossary / auto-memory に流れ込み、これらは git に乗る**。クライアント名や未公開の固有名詞がコミット履歴に永久に残るのを防ぐため、以下を必須とする。

1. **匿名トークン化を昇格時に強制**: 指摘をルール化（§4 で learnings → 確定ルールへ昇格）する瞬間に、固有名詞・クライアント名・未公開情報を匿名トークン（例: `<CLIENT_A>`, `<PRODUCT_X>`）へ置換する。生の固有名詞のまま git に乗せない。
2. **置換は承認ゲートと同時に人が確認**: §4 の差分承認のステップで、マスキング漏れがないかを人が目視する。
3. **glossary の取り扱い**: 実名⇔匿名トークンの対応表が必要な場合は、対応表自体を `.gitignore` 対象の別ファイル（例: `glossary/.client-map.local.json`）に置き、git に乗せない。共有する glossary は匿名トークン側のみ。
4. **§2.5 の機密度タグ**で「機密」とされた案件は、learnings 化の既定動作をマスキング必須にする。

## 4. スキル改善系（learnings loop / 更新スキル）

**目的**: 人が繰り返す指摘を取り込み、該当ジャンルの子スキルを育てる。「使うほど会社専用に賢くなる」中核機構。

### 4.0 auto-memory と learnings.md の役割分担（§M3 対策）

両者に同じ指摘が分散するとサイロ化（§1.2 が嫌う事態）するため、境界を定義する。

- **auto-memory**: セッション内の揮発的な気づき・文脈メモ。正式ルール化の前段で Claude が自動で書く「下書きメモ」。正本ではない。
- **`learnings.md`**: 承認を待つ正式キュー＋確定ルールの蓄積先。git 管理される正本。
- フロー: auto-memory（気づき）→ 人が拾って learnings.md の「承認待ち」へ → 承認 → 確定ルール／子スキルへ反映。

### 4.1 昇格の二経路（§C2 対策：初期に発火しない問題の解消）

「N 件溜まるまでルール化しない」だけだと、ジャンル×クライアント×指摘種別で母数が割れ、現実の案件量では各バケツが N に届かず**中核価値が起動しない**。そこで主経路を反転させる。

- **主経路：人が即承認すれば 1 件でも昇格**。翻訳者がプロとして「これは恒久ルールにすべき」と判断した指摘は、1 件でも承認をもって確定ルールへ昇格する。運用初期はこれが基本。
- **安全弁：N 件自動昇格**。人の明示承認がないまま同種の指摘が N 件溜まったら、システムが「ルール化候補」として提案する。これは過剰一般化を防ぐための**自動昇格の安全弁**であって、主経路ではない。
- どちらの経路でも、§3.5 のマスキングと §4.2 の統治ルールを通す。

### 4.2 統治ルール（この更新スキルの中身＝事故防止の本体）

1. **一般化の閾値**: 1 件の指摘で過剰一般化しない。**ただし人の即承認は例外**（§4.1 主経路）。自動昇格は「N 件溜まる」を満たして初めて提案。
2. **衝突チェック**: 追記前に既存規則と重複・矛盾しないか照合。矛盾時は統合 or 却下を提案。
3. **置き場の規約**: 指摘をルール化する際、必ず属するプロファイルをタグ付け。迷うものは `common/` に置く（複数スキルへの重複を防ぐ）。**昇格と同時に §3.5 のマスキングを適用**。
4. **承認ゲート**: 更新は「差分案を提示 → 人が承認 → 反映」。Code の差分レビュー UI がこれを担う。完全自動も技術的には可能だが、品質保証上ゲートを残す。

**仕組み（足場）**
- **learnings loop パターン**: セッション開始時に読み、終了時に更新する `learnings.md` に指摘を蓄積し、該当スキルの規則に反映する。
- 更新の自動化メタスキル: `skill-improve`（OIAE: Observe/Inspect/Amend/Evaluate）。蓄積ログから再発する失敗パターンを抽出し、証拠ベースの修正を提案し、過去の変更の有効性を評価して維持/差し戻しを判断する。採用判定は §8.1。

**環境差**
- Code: ファイル書き換え＋git で完全自動に閉じられる（ただしゲートは選択として維持）。
- Web/Cowork: 機械的書き戻しに明示ステップが残る＝半自動。

## 5. AI-writing（書き換え）ツール

出力を自然にする／AI 臭を除く層。**目的が「品質向上」のものだけを採用し、「検出回避」目的のツール（§6）とは切り分ける。**

| ツール | 性格 | 適性ジャンル | 形態 |
|---|---|---|---|
| `conorbronsdon/avoid-ai-writing` | 42 パターン＋109 語置換表＋構造検出。最も網羅的・厳格 | 広報・読み物 | SKILL.md（多言語派生あり） |
| `blader/humanizer` | Wikipedia ベース、personality/soul を注入 | 音声台本・コピー | 単一 SKILL.md（Web Upload 可） |
| `matsuikentaro1/humanizer_academic` | 学術・医学論文向け適応。formal な格を壊さず学術 AI 臭のみ除去 | 学術・申請 | SKILL.md（blader 派生） |
| `brandonwise/humanizer` | CLI＋統計エンジン（burstiness/文長分散/n-gram、0-100 スコア、フォルダ一括、退行検知、版間比較） | 全ジャンル横断 QA ゲート | CLI（npm） |
| `avoid-ai-writing-ja`（ローカル導入済み候補） | 日本語 AI-ism 検出・書き直し。voice profile（カジュアル／プロフェッショナル／テクニカル／親身／率直）、detect-only/edit-in-place、収束反復対応 | 日本語ナレ原稿・日本語成果物 | Skill（実機検証で採否判定） |
| `avoid-ai-writing-multilingual` | ネイティブ言語研究に基づく多言語派生（DE/RO、FR/ES 計画） | 非英語ターゲット時／日本語版自作の手本 | SKILL.md |

**配線の原則**
- 契約・公文書には書き換え系を**当てない**（逐語性が命）。当てても detect モードの参考まで。
- **日本語 AI 臭の処理は、まず既存の `avoid-ai-writing-ja` を実機検証してから自作可否を決める（§H1 対策）。** 英語特化ツールを日本語にそのまま当てない。`avoid-ai-writing-ja` が日本語 AI 臭（「いかがでしたか」「〜と言えるでしょう」「ポイントは 3 つ」「体言止め連続」等）を十分カバーするなら自作スコープを削れる。不足分だけ multilingual の手法（翻訳ではなくネイティブ研究）で補う。
- 会社レベルの品質ゲートとして brandonwise CLI の退行検知（ベースライン比較）を CI 的に使うと、多数ファイルを定量管理できる。

> **自作スコープの縮小方針（§H1）**: 「日本語ツールは自作」と決め打ちせず、§9-1 の起草前に `avoid-ai-writing-ja` を §8.1 基準で検証する。SPEC の「自作せず構造を借りる」方針（§2 参考実装）と整合させる。

## 6. 検出系

クライアントが納品物を AI 判定にかけるリスクを測る層。

| ツール | 性格 |
|---|---|
| Pangram | 偽陽性率が最も低い（独立研究で長文ほぼゼロ、厳格上限を満たす）。高ステークス向き |
| GPTZero | 知名度高・無料枠。ただし独立テストで偽陽性が出やすい |
| Originality.ai | パラフレーズ済みに強い、プレミアム。出版・SEO 向き |
| Copyleaks | 30+ 言語対応で多言語向き |

**翻訳会社にとって致命的な前提（必読）**
- **翻訳された英語は、検出器が最も誤爆しやすいプロファイルそのもの。** 非ネイティブ／きれいで形式的な文体は「予測可能すぎる」として誤判定されやすい（Stanford 研究で非ネイティブ英語エッセイの 61% が AI 誤判定）。
- パラフレーズ/humanize 済みは検出精度が大きく落ち、単一スコアを証拠にしてはならない。複数で照合する。
- 帰結: 検出器は**合否判定ではなくリスクレーダー**として扱う。防御は「スコアを通す」ことより、**過程の記録（下訳履歴・訳者ノート）**で人間関与を示すこと。

### 6.2 運用形態（§H2 対策：誰が・いつ・どう実行するかを定義）

Pangram 等は Web サービスで、Code 内から自動実行できない。よって検出系を独立した「ツール」として宙に浮かせず、**運用手順を子スキルに埋め込む**。

- **広報・読み物プロファイルのチェックリスト項目（手動・任意）**として実装する。子スキルが「（任意）納品前に検出器でリスク確認：手順 → 結果を訳者ノートに記録」と指示する。
- **実行者と記録**: 担当翻訳者が手動でコピペ実行し、スコアと使用ツール・日時を訳者ノート（過程の記録）に残す。これが §6 の「人間関与を示す防御線」と接続する。
- 自動化しない（できない）ことを仕様として明記し、未実装の自動連携を前提にした設計にしない。

### 6.3 設計上の鉄則

- **検出チェックを学術・申請／契約のパイプラインに組み込まない。** 「検出を通すために書き換える」ループは忠実性を削り、特に申請書で致命的。
- 検出は本流の外に置く独立 QA レーンとし、必要なジャンル（広報等）でのみ §6.2 の手動手順でリスク確認に使う。

**回避系（"undetectable" web ツール）について**
- Phrasly / AI Busted / Undetectable 系は「検出を通す」ことが目的で、品質を落とし、納品物を偽装する方向に倒れやすい。信用商売の翻訳会社では中核に据えない。**書き換え系（品質向上）と回避系（判定すり抜け）は目的が別物**として明確に分離する。

## 7. 永続化・共有・運用

- **永続化**: 育てる対象は書込可ファイル（`.claude/skills/` / Project 作業ファイル）。これを git 管理。
- **共有**: リポジトリ共有で二人運用。`williamzujkowski` 系の config 同期や設定バックアップ（settings/skills/agents/memory を private repo に同期、シークレットスキャン・競合検出付き）も検討。採用判定は §8.1。
- **育成の単一正本運用（§M1 対策）**: Team プランの共有スキルは閲覧専用＝安定版配布向き。よって**育成（書込）作業は「正本担当 1 名」に集約し、もう 1 名は安定版を pull するだけ**とする。二人が同時に learnings を育てるとマージ衝突が起き、config 同期の競合検出では「翻訳ルールの意味的マージ」までは解決できないため。役割（誰が正本担当か）を運用開始前に確定する。
- **プラン**: 個人 Pro ×2 だと共有に手作業が残る。育つライブラリを二人で回すなら **Team プラン**でスキル共有が素直になる（共有スキルは閲覧専用＝安定版配布向き）。判断は §9-4。
- **機密**: データ学習を設定でオフ。クライアント名・固有名詞のマスキング運用（§3.5）。機密原稿は `.gitignore` で追跡しない。

## 8. セキュリティ

- 三者製スキルは任意コード／プロンプトインジェクションの面を持つ。**導入前に SKILL.md を精読し、スキャンする。**
- 機密原稿を扱う環境に未検証スキルを入れない。まず `humanizer_academic` と `brandonwise/humanizer`（本用途に最も効く 2 つ）＋日本語向けの `avoid-ai-writing-ja` から実機検証する。

### 8.1 外部ツール採用の判定基準（§H3 対策）

§5/§6/§7 の外部スキル・リポジトリは、採用前に以下を**判定テーブルとして必ず記録**する。SPEC 末尾の「採用前に確認」を運用に落とし込む。

| 判定軸 | 確認内容 | 不可なら |
|---|---|---|
| 実在性 | リポジトリ／スキルが現存するか | 候補から除外 |
| ライセンス | 再配布可否（二人運用・git 共有 §7 に直結） | 再配布不可なら共有設計を見直し |
| 保守性 | 最終コミット日・更新頻度 | 長期放置は採用リスクとして明記 |
| 健全性 | star 数・利用実績・issue 状況 | 低実績は「自作 or fork して内製化」を検討 |
| 安全性 | SKILL.md 精読・スキャン（§8 冒頭） | インジェクション面があれば除外 |

> 個人リポジトリは消失・破壊リスクがあるため、採用するものは fork してリポジトリ内 `tools/` に取り込み、外部消失に備える。
