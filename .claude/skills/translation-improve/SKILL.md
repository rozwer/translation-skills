# translation-improve — 改善ループ（learnings loop / 更新スキル）

SPEC.md §4 の中核機構。人が繰り返す指摘を取り込み、該当ジャンルの子スキルを育てる。
「使うほど会社専用に賢くなる」を成立させる本体。正本ログは `improvement/learnings.md`。

## 0. このスキルの責務

- セッション内で出た翻訳者の指摘を拾い、`learnings.md` §A（承認待ちキュー）へ記録する。
- 人の承認を受けたものを §B（確定ルール）へ昇格し、該当スキルへ反映する。
- 反映時に固有名詞のマスキングと衝突チェックを必ず通す（事故防止）。

反映先の例:
- `profile-*/SKILL.md`（音声台本・広報・学術申請・契約公文書など、ジャンル固有ルール）
- `common/house-style.md`（横断ルール・境界ケース）

肥大させない。判断と手順だけを置き、規則の実体は反映先ファイルに書く。

## 1. auto-memory と learnings.md の役割分担（SPEC §4.0）

| 置き場 | 性格 | 正本か |
|---|---|---|
| auto-memory | セッション内の揮発的な気づき・文脈メモ。Claude が自動で書く下書き | いいえ |
| `learnings.md` §A | 承認を待つ正式キュー | はい（git 管理） |
| `learnings.md` §B | 承認・反映済みの確定ルール台帳 | はい（git 管理） |

フロー: 気づき（auto-memory）→ §A 承認待ちへ → 人が承認 → §B へ昇格 → 該当スキルへ反映。
同じ指摘を両方に分散させない。正式化したら auto-memory 側は下書きとして扱い、§A を起点にする。

## 2. 昇格の二経路（SPEC §4.1・最重要）

- **主経路（人承認）**: 翻訳者がプロとして「これは恒久ルールにすべき」と判断した指摘は、**1 件でも承認をもって昇格する**。運用初期はこれが基本。
- **安全弁（N 件自動）**: 人の明示承認がないまま同種の指摘が N 件溜まったら、システムが「ルール化候補」として提案する。これは過剰一般化を防ぐための**自動昇格の安全弁であって、主経路ではない**。

> N 件待ちを既定にすると、ジャンル×クライアント×指摘種別で母数が割れ、各バケツが N に届かず中核価値が起動しない。だから人承認を主経路に置く。

どちらの経路でも §3 のマスキングと §4 の統治ルールを必ず通す。

## 3. マスキング強制（SPEC §3.5）

`learnings.md` / `glossary/` / auto-memory に**書き出す瞬間**に、固有名詞を匿名トークンへ置換する。

| 種別 | トークン例 |
|---|---|
| クライアント企業・団体名 | `<CLIENT_A>` |
| 製品・サービス・プロジェクト名 | `<PRODUCT_X>` |
| 個人名 | `<PERSON_1>` |
| 未公開の固有情報（金額・日付・場所） | `<CONF_n>` |

- 生の固有名詞のまま git に乗せない。実名⇔トークン対応表は `.gitignore` 済みのローカルファイルに置く。
- マスキング漏れは §4-④ の承認ゲートで**人が目視で確認**する。
- ルーター設問5で「機密」とされた案件は、マスキングを既定で必須にする。
- 詳細表は `common/house-style.md` §1 を参照（実体はそちらに一元化）。

## 4. 統治ルール（事故防止の本体・SPEC §4.2）

ルール化のたびに、以下4点を順に通す。

| # | ルール | 内容 |
|---|---|---|
| ① | 一般化の閾値 | 1 件の指摘で過剰一般化しない。**ただし人の即承認は例外**（主経路）。自動昇格は N 件を満たして初めて提案。 |
| ② | 衝突チェック | 追記前に反映先の既存規則と重複・矛盾しないか照合。矛盾時は統合 or 却下を**提案**し、勝手に上書きしない。 |
| ③ | 置き場の規約 | 必ず属するプロファイルをタグ付け。迷うものは `common/house-style.md` へ（複数スキルへの重複を防ぐ）。昇格と同時に §3 のマスキングを適用。 |
| ④ | 承認ゲート | 「差分案を提示 → 人が承認 → 反映」。Code の差分レビュー UI がこれを担う。完全自動は技術的に可能だが、品質保証上ゲートを残す。 |

## 5. learnings loop の運用手順

### 5.1 セッション開始時
1. `improvement/learnings.md` を読む。§B の確定ルールを当該セッションの作業基準として把握する。
2. §A に未処理の承認待ちがあれば、今回の案件と関係するものを頭に入れておく。

### 5.2 セッション中
3. 翻訳者の修正・指摘が出たら、auto-memory に気づきとして書き留める（下書き）。

### 5.3 セッション終了時（このスキルの本番）
4. 今回の気づきを棚卸しし、ルール化候補を選ぶ。
5. 各候補を §3 のトークンでマスキングし、`learnings.md` §A に1件1ブロックで追記する。フォーマットは §A 冒頭のコメント雛形に厳密に合わせる:
   - `指摘 / 文脈 / 提案ルール / 置き場候補 / 状態（承認待ち）/ 昇格経路（人承認 or N件）`
6. 翻訳者が「恒久ルールにすべき」と判断したものについて、統治ルール①〜③を通したうえで**差分案を提示**する（反映先ファイルの追記 diff ＋ §B への新規 RULE エントリ）。
7. 人が承認したら反映を実行する:
   - 反映先（`profile-*/SKILL.md` の該当節 / `common/house-style.md`）へ規則文を追記。
   - `learnings.md` の §A 該当ブロックを §B へ移し、`反映先のファイルとセクション`・`承認日`・`根拠`・`評価（空欄で開始）` を記入。RULE 番号を採番（`RULE-NNNN` は 4 桁ゼロパディング。最初の確定ルールは `RULE-0001`）。
   - 同一指摘が §A に複数あれば集約して 1 ルールにまとめる。
8. 承認されなかった候補は §A に残す。同種が N 件溜まったら §B 安全弁として「ルール化候補」を改めて提案する。

## 6. OIAE フレームワーク（内製・skill-improve は採用見送り）

蓄積ログから改善を回す考え方。外部スキル `skill-improve` は正本リポジトリを特定できず検証不能のため**採用見送り**とし（`tools/ADOPTION.md` 参照）、OIAE の発想だけを本スキルに内装する。
OIAE の各段を本ループに重ねて使う:

- **Observe**: §A/§B の蓄積ログから再発する指摘パターンを抽出する。
- **Inspect**: 該当する反映先スキルの現規則と突き合わせ、抜け・矛盾を特定する（統治ルール②と接続）。
- **Amend**: 証拠（intake 件数・人承認）に基づき修正 diff を提案する。承認ゲートを必ず通す。
- **Evaluate**: 過去に反映したルールが有効だったかを §B の `評価` 欄で追跡し、退行があれば差し戻し（該当規則を撤回し §A へ戻す）を提案する。

## 7. 環境差（SPEC §4 末尾）
- Code: ファイル書き換え＋git で更新ループが完全自動に閉じる（ゲートは選択として維持）。
- Web / Cowork: 機械的な書き戻しに明示ステップが残る＝半自動。育てる対象は書込可ファイルに限る。
