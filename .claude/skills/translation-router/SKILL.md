# translation-router

翻訳・ナレーション案件の入口。ツールを起動する前に「何のジャンルか／どのパイプラインで処理すべきか」を確定させる。Claude の暗黙の推測（誤選択しやすい）を、使用者の明示選択を主経路とする決定的トリアージに置き換えるのが役割。

このスキルは判定と振り分けだけを行う。実際の書き換え・チェックは各プロファイルスキルが担当する（単一責務・progressive disclosure）。

## 起動タイミング

日本語→英語の翻訳、クライアントが作った英訳（AI 使用）の修正、日本語ナレ原稿の推敲 — このいずれかの依頼が来たら、他のツールより先にこのスキルを動かす。

## プロファイル対応表（決定的マッピング）

設問1の選択肢 ID をそのままプロファイル ID にマップする。LLM の推論を挟まない。同じ選択は必ず同じプロファイルへ向かう。

| ID | 成果物の種類 | 振り分け先スキル |
|----|------------|----------------|
| 1 | 音声台本（アナウンス・ナレーション） | `profile-audio-script` |
| 2 | 広報・読み物 | `profile-editorial` |
| 3 | 学術・申請（論文・科研費など） | `profile-academic-grant` |
| 4 | 契約・公文書 | `profile-contract-legal` |
| 5 | その他／わからない | フォールバック（下記）。自動確定しない |

## インタビュー設問（5問）

タスク開始時に必ず順に確認する。使用者は翻訳のプロだが AI は初学者なので、選択肢を明示して聞く。

1. **成果物の種類** — 台本／広報・読み物／学術・申請／契約・公文書／その他。この答えが決定的にプロファイルを決める。
2. **忠実さの要求** — 意図に忠実でよい（transcreation 可）↔ 逐語・定訳を厳守。
3. **制約** — 尺・文字数、定訳の有無、締切、分野。
4. **人間レビューの要否** — クライアント要件、機密度から判断。
5. **機密度タグ** — クライアント名・固有名詞を含むか／公開可否。改善ループでの匿名トークン化判断に使う。

## 判定手順

### 主経路（決定的・既定）

1. 設問1の選択（1〜4）を上の表でプロファイル ID に変換する。LLM 推論はしない。
2. 該当プロファイルスキルと、そのプロファイルが使う書き換えツールを選ぶ。
3. `route` は `main` として記録する。

### フォールバック経路（設問1が「5: その他/わからない」のときだけ）

1. 原稿の特徴（文体・用途・読者・定訳の有無）から候補プロファイルを **提示** する。
2. **自動確定はしない。** 最終決定は使用者が行う。使用者が選んだ ID を採用する。
3. `route` は `fallback` として記録する。後追いで判定を見直せるようにするため。

### 境界ケースの扱い

台本と読み物の中間など、どちらか一方へ寄せきれない原稿は、無理に片方へ寄せない。判定の不確実性を **不確実フラグ**（`boundary_flag: true`）として残したまま、`common/house-style.md` の横断ルールで処理する。プロファイル固有の作法を強く当てず、共通レイヤーを優先する。

## 実行前宣言

振り分けが決まったら、ツールを動かす前に次を宣言する。

- 使うプロファイル（例: `profile-editorial`）
- 経路（`main` か `fallback` か）
- 機密度タグの内容と、それに伴う注意（機密なら改善ループでマスキング必須）
- 不確実フラグの有無

## ログ追記（監査用）

宣言と同時に `improvement/router-log.jsonl` へ1タスク1行で追記する。フィールド例:

```json
{"timestamp": "2026-06-17", "profile": "profile-editorial", "route": "main", "confidentiality": "client_name_included", "boundary_flag": false}
```

- `timestamp`: プログラムから自動取得はできないので、Claude が現在日付（環境が示す今日の日付）を補うか、使用者が記入する。
- 1行が壊れた JSON にならないよう、追記は1行で完結させる。
- フォールバック経由（`route: "fallback"`）の判定は、後で人がまとめて見直せるよう必ず残す。

## 参照する共有ファイル

- `common/house-style.md` — 横断ハウススタイル・共通 NG・境界ケースの規則。不確実フラグを立てた案件はここで処理する。
- `improvement/learnings.md` — 蓄積された確定ルールと承認待ちキュー。振り分け先プロファイルに関係するルールがあれば、プロファイルスキルが参照する。

## タスク終了時

成果物が固まったら、改善ループへ誘導する。

- 今回の指摘・修正で恒久ルールにすべきものがあるか問いかける。
- ルール化する場合は `translation-improve` スキルを案内する。プロとして「これは恒久ルール」と判断した指摘は、人の承認をもって1件でも昇格できる（主経路）。
- 機密度タグが「機密」の案件は、ルール化時に固有名詞・クライアント名を匿名トークン（例: `<CLIENT_A>`）へ置換する必要がある旨を添える。生の固有名詞のまま履歴に残さない。

## 参考実装の足場

ルーター構造は `hussi9/skill-router`（1枚の SKILL.md で多問トリアージ＋宣言＋jsonl ログ＋監査スクリプト）の構成を借りて本用途で検証中。外部ツールの採用可否や配線の詳細は別途整備する `tools/ADOPTION.md` を参照する。
